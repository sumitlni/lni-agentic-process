# tools/status — one source of truth, generated views, and a checker

Three files, stdlib Python only, no dependencies. They implement §9 of `BUILD_PRACTICES.md`. I wrote
these after my own tracker grew into a 2,504-line file that was quietly lying to me — the checker is
the part that matters, not the dashboard.

```
status.py      serve the dashboard · --check (validate) · --render (regenerate the board)
STATUS.html    the dashboard (click a badge to change a status)
status.json    THE SOURCE OF TRUTH — you create this (see the schema below)
queue.md       GENERATED by --render. Do not edit.
```

**Nothing here is project-specific.** `project`, `subtitle`, and `modules` come from `status.json`, so
these files drop into any repo unchanged.

## Install

Put `status.py` + `STATUS.html` in a process folder one level below your repo root (e.g. `Strategy/`
or `process/`) and create `status.json` beside them. The script treats its own folder as the board and
its parent as the workspace root, so `ref` paths resolve relative to it.

```
python3 status.py                        # serve the dashboard (default), opens a browser
python3 status.py --check                # validate; NON-ZERO EXIT on any error
python3 status.py --render               # regenerate queue.md
python3 status.py --render --check-sync  # fail if queue.md is stale vs the JSON
python3 status.py --editor Code          # which editor the "open" links use (or set ABP_EDITOR)
```

You can also leave the script where it is and point it at a board:

```
python3 tools/status/status.py --dir Strategy --check
```

Try it right now against the worked example:

```
python3 tools/status/status.py --dir example/Strategy --check
python3 tools/status/status.py --dir example/Strategy --render
```

Wire `--check` into a git hook, CI, or your agent's stop hook. It exits non-zero so it can gate.

## Schema

Top level: `project`, `subtitle`, `last_updated` (written by the script), `modules[]`, `items[]`.

Each item:

| field | meaning |
|---|---|
| `id` | short unique key (`UI-7`, `DEPLOY-CENTS`). Used by `blocked_by`. |
| `module` | must be one of `modules[]` — `--check` enforces it. |
| `type` | **`round`** = work (ephemeral: leaves the rendered board when done). **`ops`** / **`launch`** = a gate (durable: human/external, no commit records it, stays visible until done). **`doc`** = a coordinator job. |
| `title` | one line. |
| `status` | `pending` · `active` · `done` · `deferred`. |
| `owner` | who does it (`code`, `you`, `coordinator`…). |
| `note` | the prose that matters — why it's parked, the hazard on a gate. **Rendered**, never retyped into markdown. |
| `ref` | path (relative to this folder) to the detail: a prompt file, the ledger, a doc. |
| `sec` | a section label inside `ref`; the dashboard's "open" link jumps to it by scanning, so no stored line numbers to drift. |
| `blocked_by` | `[ids]`. **Blocked is DERIVED from this, never stored** — so it can't go stale. |
| `drafted` | for a round: is the prompt actually written? A drafted round whose `ref` file doesn't exist is an **error**. |

## What `--check` catches

This is the load-bearing part — a tracker nobody verifies is a rumour.

- a `blocked_by` id that doesn't exist, and dependency **cycles**
- an item `active` while blocked, or `done` while still blocked
- a **drafted round whose prompt file isn't on disk**
- a `deferred` item with **no reason** in `note` (if you can't say why it waits, it isn't real work)
- duplicate ids, unknown `module`, bad `status`
- a presence-board row saying `IN PROGRESS` with no `active` round — *and the reverse*, which is the
  drift nobody thinks to look for

`--check-sync` then guarantees the rendered board matches the source. A generated file nobody
regenerates is just a stale file with extra steps.

**Try to break it.** Feed it a dangling blocker, a cycle, a parked item with no reason, and confirm it
exits non-zero on each. A validator you haven't tried to break is just another unverified file.

## The rendered board

`--render` writes `queue.md`, grouped by **state, not epic**: LIVE / READY / BLOCKED / GATES /
BACKLOG / PARKED / coordinator jobs. Done rounds are **excluded from the view** but **retained in the
JSON** — that's the history, and for a gate it's the only record that exists.

Commit `queue.md` so agent sessions read it at boot without running anything. A status change from the
dashboard re-renders it automatically, so the two can't drift.

## Notes

- The "open in editor" links use macOS `open -a` (or the editor's CLI). Everything else is portable.
- The dashboard **refuses to mark a blocked item `active`**. It shouldn't let you lie to yourself
  about what you can start.
